## 1. 类的基本定义

> **🤔 为什么需要类？**
> 函数解决了"怎么做"的问题，但现实世界里东西是有"状态"的。一只狗有名字、年龄（状态），也会叫、跑（行为）。类就是把"状态+行为"打包成一个整体。
>
> **🎯 类比理解**
> 类 = 饼干模具，实例 = 用模具压出来的饼干。模具定义了形状（结构），每块饼干可以有不同口味（属性值）。`__init__` 就是"往模具里填料"的过程。
>
> **💡 设计思路**
> OOP 的核心思想：**用代码模拟现实世界**。把相关的数据和操作放在一起，而不是散落在各处。这样代码的组织方式更接近人类思维——我们想到"狗"，自然联想到它的属性和行为。
>
> **🔄 举一反三**
> - 游戏开发：Player 类有 hp/attack/position + move()/attack()
> - Web 开发：User 类有 name/email + login()/logout()
> - 思考：什么时候该用类，什么时候函数就够了？当你发现"一组函数总是操作同一批数据"时，就该考虑类了

```python
class Dog:
    def __init__(self, name, age):
        self.name = name  # 实例属性
        self.age = age
    
    def bark(self):
        return f"{self.name}说: 汪汪汪!"

dog1 = Dog("旺财", 3)
print(dog1.bark())  # 旺财说: 汪汪汪!
```

> **💡 `__init__` vs `__new__` 的思路**
> `__new__` 创建实例（分配内存），`__init__` 初始化实例（填充数据）。
> 类比：`__new__` = 盖房子（建筑结构），`__init__` = 装修（放家具）。

## 2. 实例属性与类属性

> **🤔 为什么要区分这两种属性？**
> 有些信息是"大家共享的"（学校名），有些是"每人不同的"（名字）。类属性 = 共享白板，实例属性 = 个人笔记本。
>
> **🎯 类比理解**
> - 类属性 = 教室里的投影仪（全班共用一个）
> - 实例属性 = 每个人的课本（各自不同）
>
> **💡 设计思路**
> Python 的属性查找是**自下而上**的：先找实例，再找类。这就像你先翻自己笔记本，没有再看共享白板。
>
> **🔄 举一反三**
> - 计数器（统计创建了多少个实例）→ 用类属性
> - 配置信息（所有实例共享）→ 用类属性
> - 每个用户的余额 → 必须是实例属性

```python
class Student:
    school = "Python大学"  # 类属性 - 共享
    count = 0
    
    def __init__(self, name):
        self.name = name   # 实例属性 - 独有
        Student.count += 1

s1 = Student("小明")
s2 = Student("小红")
print(Student.count)  # 2
```

### `__slots__` 限制属性

> **💡 思路：为什么要限制属性？**
> 默认情况下 Python 对象用 `__dict__` 存属性（灵活但费内存）。`__slots__` 告诉 Python "只会有这几个属性"，从而优化内存。
> 类比：`__dict__` = 弹性书包（什么都能装），`__slots__` = 固定格子的工具箱（空间省，但只能放指定工具）。

```python
class Point:
    __slots__ = ('x', 'y')  # 只允许 x, y
    def __init__(self, x, y):
        self.x, self.y = x, y
# p.z = 3  # ❌ AttributeError
```

## 3. 方法类型

> **🤔 为什么要有三种方法？**
> 不同方法的"视角"不同：
> - 实例方法：站在"某个具体对象"的角度（我这只狗叫什么）
> - 类方法：站在"整个类"的角度（总共有多少只狗）
> - 静态方法：跟类相关但不需要访问类或实例的工具函数
>
> **🎯 类比理解**
> - 实例方法 = 员工的工作（需要知道"我是谁"→ self）
> - 类方法 = HR 的工作（需要知道"公司"→ cls，不关心具体某个人）
> - 静态方法 = 公司食堂的微波炉（放在公司里，但谁都能用，不需要身份）
>
> **💡 设计思路**
> 这三种方法体现了**最小权限原则**：方法需要什么信息，就给什么信息。不需要访问实例就别给 self，不需要访问类就用 staticmethod。权限越小，代码越安全、越容易测试。

```python
class Date:
    def __init__(self, year, month, day):
        self.year, self.month, self.day = year, month, day
    
    def display(self):           # 实例方法：需要 self
        return f"{self.year}-{self.month}-{self.day}"
    
    @classmethod
    def from_string(cls, s):     # 类方法：需要 cls（替代构造器）
        y, m, d = map(int, s.split('-'))
        return cls(y, m, d)
    
    @staticmethod
    def is_valid(y, m, d):       # 静态方法：纯工具，不需要 cls/self
        return 1 <= m <= 12 and 1 <= d <= 31
```

> **🔄 举一反三**
> - `@classmethod` 最常见用途：替代构造器（多种创建方式）
> - `@staticmethod` 适合：放在类里只是为了逻辑归类（不用也行，但放着更整齐）
> - 思考：如果一个方法既不用 self 也不用 cls，先考虑它是不是该改成模块级函数

## 4. 继承

> **🤔 为什么需要继承？**
> 现实中事物有层次关系："狗是动物"，"学生是人"。继承让你复用已有代码，只写"不同的部分"。
>
> **🎯 类比理解**
> 继承 = 子承父业。儿子继承了父亲的家产（属性和方法），但可以有自己的新技能（新方法），也可以把家传手艺改良（方法重写）。
>
> **💡 设计思路**
> 继承的核心思想：**DRY (Don't Repeat Yourself)**。共性提取到父类，差异放在子类。但继承也有代价——父子耦合太紧。现代设计倾向"组合优于继承"。
>
> **🔄 举一反三**
> - Django 的 View/Model 继承体系
> - 游戏里的 Character → Warrior/Mage/Archer
> - 思考：如果继承链太深（A→B→C→D→E），出了 bug 怎么定位？这就是"组合优于继承"的原因

```python
class Animal:
    def __init__(self, name):
        self.name = name
    def speak(self):
        raise NotImplementedError

class Dog(Animal):
    def speak(self):
        return f"{self.name}: 汪汪!"

class Cat(Animal):
    def speak(self):
        return f"{self.name}: 喵喵~"
```

### super() 的思路

> **💡 为什么要用 super() 而不是直接写父类名？**
> 因为在多继承中，直接写父类名会导致某些父类被调用多次或跳过。super() 遵循 MRO（方法解析顺序），保证每个类只被调用一次。

```python
class Child(Base):
    def __init__(self, x, y):
        super().__init__(x)  # 让 Python 按 MRO 找到正确的父类
        self.y = y
```

### 多继承与 MRO

> **💡 设计思路：Python 如何解决"菱形继承"问题？**
> C3 线性化算法。查找顺序是广度优先+保证子类在父类前面。用 `Class.__mro__` 可以看到具体顺序。

## 5. 多态

> **🤔 为什么需要多态？**
> 你写了一个函数处理"形状"，不管是圆、方、三角，都调用 `.area()`。多态让你写出**不关心具体类型**的通用代码。
>
> **🎯 类比理解**
> 多态 = USB 接口。不管你插的是鼠标、键盘还是U盘，电脑只需要"读取USB设备"这一个接口。设备不同，行为不同，但接口统一。
>
> **💡 设计思路**
> Python 的多态基于**鸭子类型**："如果它走路像鸭子、叫声像鸭子，那它就是鸭子。"不需要严格的继承关系，只要有同名方法就能多态。这比 Java 的严格类型检查更灵活。
>
> **🔄 举一反三**
> - 任何有 `__len__` 方法的对象都能用 `len()`
> - 任何有 `__iter__` 方法的对象都能用 for 循环
> - 思考：Python 的 `for x in obj` 不关心 obj 是列表、元组还是文件——这就是多态

```python
shapes = [Rectangle(3, 4), Circle(5), Triangle(6, 8)]
for shape in shapes:
    print(f"{shape.__class__.__name__}: {shape.area():.2f}")
# 同一个 .area() 调用，不同的计算逻辑
```

## 6. 封装与访问控制

> **🤔 为什么需要封装？**
> 你买手机不需要知道CPU怎么工作，只需要按屏幕。封装隐藏内部复杂性，只暴露必要的接口，防止外部"乱改内部零件"。
>
> **🎯 类比理解**
> - `public` (无前缀) = 商店橱窗（谁都能看和用）
> - `_protected` (单下划线) = 仓库（员工能进，写着"非请勿入"但门没锁）
> - `__private` (双下划线) = 保险柜（真的锁了，但撬锁还是能打开）
>
> **💡 设计思路**
> Python 的哲学是**"我们都是成年人"**。没有真正的 private，只有"约定"。单下划线是温和提醒"别碰"，双下划线是名称改写（防止意外覆盖），但不是不可访问。这体现了 Python 信任程序员的文化。
>
> **🔄 举一反三**
> - API 设计时：用户能调用的方法不加下划线，内部辅助方法加 `_`
> - `__` 最常见用途：防止子类不小心覆盖父类的内部属性
> - 思考：Java 的 private 是编译器强制的，Python 是约定+名称改写——哪种更好？

```python
class BankAccount:
    def __init__(self, owner, balance=0):
        self.owner = owner       # 公开
        self._balance = balance  # 约定：别直接改
        self.__pin = "1234"      # 名称改写 → _BankAccount__pin
    
    def deposit(self, amount):   # 通过方法操作，可以加验证
        if amount > 0:
            self._balance += amount
```

## 7. 属性装饰器 property

> **🤔 为什么需要 property？**
> 有时你想让"方法调用"看起来像"属性访问"。比如温度对象，你想写 `t.fahrenheit` 而不是 `t.get_fahrenheit()`。property 让接口更自然。
>
> **🎯 类比理解**
> property = 智能家居开关。看起来只是普通开关（像属性），但按下去背后有一套逻辑（像方法）——可以验证、计算、触发其他动作。
>
> **💡 设计思路**
> property 的核心价值：**把实现细节和接口分离**。今天 `obj.x` 直接读属性，明天你想加个验证逻辑，只需要加 property，调用者的代码一行不用改。这就是"开闭原则"。
>
> **🔄 举一反三**
> - 计算属性：`circle.area` 根据 radius 实时计算
> - 验证属性：`user.age = -1` 时自动拒绝
> - 只读属性：只有 getter 没有 setter

```python
class Temperature:
    def __init__(self, celsius=0):
        self._celsius = celsius
    
    @property
    def celsius(self):
        return self._celsius
    
    @celsius.setter
    def celsius(self, value):
        if value < -273.15:
            raise ValueError("不能低于绝对零度!")
        self._celsius = value
    
    @property
    def fahrenheit(self):  # 只读计算属性
        return self._celsius * 9/5 + 32

t = Temperature(25)
print(t.fahrenheit)  # 77.0 — 像属性一样访问，背后是计算
```

## 8. 魔术方法 (Dunder Methods)

> **🤔 为什么需要魔术方法？**
> 你希望自己的类能像内置类型一样使用：`obj1 + obj2`、`len(obj)`、`for x in obj`。魔术方法就是让自定义类"融入" Python 语言的接口。
>
> **🎯 类比理解**
> 魔术方法 = 协议/合同。Python 说"如果你实现了 `__len__`，我就允许对你用 `len()`"。就像一个俱乐部的入会条件——满足条件就能享受权益。
>
> **💡 设计思路**
> Python 的"协议"设计：不用继承特定接口，只要实现特定方法就能获得对应功能。这就是**鸭子类型**的体现。比 Java 的 implements Comparable 更灵活。
>
> **🔄 举一反三**
> - 想让对象能用 `+`？实现 `__add__`
> - 想让对象能用 `for` 遍历？实现 `__iter__` + `__next__`
> - 想让对象能用 `with`？实现 `__enter__` + `__exit__`
> - 想让对象能像函数一样调用？实现 `__call__`

### 字符串表示

```python
class Point:
    def __init__(self, x, y):
        self.x, self.y = x, y
    def __repr__(self):  # 开发者看的（可复制粘贴重建对象）
        return f"Point({self.x}, {self.y})"
    def __str__(self):   # 用户看的（友好展示）
        return f"({self.x}, {self.y})"
```

### 运算符重载

```python
class Vector:
    def __init__(self, x, y):
        self.x, self.y = x, y
    def __add__(self, other):
        return Vector(self.x + other.x, self.y + other.y)
    def __mul__(self, scalar):
        return Vector(self.x * scalar, self.y * scalar)
    def __abs__(self):
        return (self.x**2 + self.y**2) ** 0.5
    def __repr__(self):
        return f"Vector({self.x}, {self.y})"

v1 = Vector(3, 4)
print(v1 + Vector(1, 2))  # Vector(4, 6)
print(abs(v1))             # 5.0
```

### 容器协议

```python
class Deck:
    def __init__(self):
        self.cards = [f"{r}{s}" for s in "♠♥♦♣" for r in range(1,14)]
    def __len__(self):           return len(self.cards)
    def __getitem__(self, i):    return self.cards[i]
    def __contains__(self, c):   return c in self.cards
```

### 上下文管理器与可调用对象

```python
# __enter__ + __exit__ → 支持 with 语句
# __call__ → 实例可以像函数一样调用
class Adder:
    def __init__(self, n):
        self.n = n
    def __call__(self, x):
        return self.n + x

add5 = Adder(5)
print(add5(3))  # 8 — 对象当函数用
```

## 9. 抽象类与接口

> **🤔 为什么需要抽象类？**
> 你想定义一个"合同"：所有形状**必须**有 area() 方法。但你不想在基类里实现它（因为基类不知道具体形状）。抽象类就是"只画蓝图不盖房"。
>
> **🎯 类比理解**
> 抽象类 = 考试大纲。大纲规定了"必须考这些题"（抽象方法），但具体答案由每个学生自己写（子类实现）。不答题就不能毕业（不实现抽象方法就不能实例化）。
>
> **💡 设计思路**
> 抽象类解决了"接口约束"问题：确保所有子类都有统一的方法签名。这在团队协作中很重要——你定义好接口，别人实现具体逻辑，调用者不用关心具体是谁。
>
> **🔄 举一反三**
> - 插件系统：定义抽象基类 Plugin，所有插件必须实现 `execute()`
> - 策略模式：定义抽象 Strategy，不同算法各自实现
> - 思考：Python 有 ABC，但很多时候直接用鸭子类型就够了——什么时候需要显式的 ABC？

```python
from abc import ABC, abstractmethod

class Shape(ABC):
    @abstractmethod
    def area(self):
        pass
    
    @abstractmethod
    def perimeter(self):
        pass

# shape = Shape()  # ❌ 不能实例化抽象类
class Circle(Shape):
    def __init__(self, r): self.r = r
    def area(self): return 3.14 * self.r ** 2
    def perimeter(self): return 2 * 3.14 * self.r
```

## 10. 元类 (Metaclass)

> **🤔 为什么需要元类？**
> 普通类控制"实例的创建过程"。元类控制"类本身的创建过程"。当你需要对所有类施加统一规则时（比如所有类都自动注册、所有类都验证属性），元类就有用了。
>
> **🎯 类比理解**
> - 类 = 饼干模具，实例 = 饼干
> - 元类 = **制造模具的机器**
> - 普通程序员用类创建对象；框架作者用元类创建类
>
> **💡 设计思路**
> Python 中一切皆对象——类本身也是对象（type 的实例）。这个设计让你能用代码"编程地创建类"。元类是 Python 反射能力的终极体现。
>
> **🔄 举一反三**
> - Django ORM 的 Model 用了元类（自动收集字段、创建数据库表）
> - 单例模式可以用元类优雅实现
> - 99% 的场景不需要元类——先考虑装饰器和 `__init_subclass__`

```python
class SingletonMeta(type):
    _instances = {}
    def __call__(cls, *args, **kwargs):
        if cls not in cls._instances:
            cls._instances[cls] = super().__call__(*args, **kwargs)
        return cls._instances[cls]

class Database(metaclass=SingletonMeta):
    pass

db1, db2 = Database(), Database()
print(db1 is db2)  # True
```

> 📚 **本节是元类的入门视角**。元类的属性查找顺序、`__init_subclass__` 替代方案、Django ORM 风格的元类应用、何时**不该**用元类，详见 **[22 章 描述符与元类深入](22_描述符与元类深入.md)**。

## 11. 数据类 dataclass

> **🤔 为什么需要 dataclass？**
> 很多类只是"数据容器"——有几个字段，需要 init/repr/eq。每次都手写这些样板代码太烦了。dataclass 自动帮你生成。
>
> **🎯 类比理解**
> dataclass = 快速表格生成器。你只需要定义"列名和类型"，它自动帮你生成"新建行"、"显示行"、"比较行"等功能。
>
> **💡 设计思路**
> 减少样板代码（boilerplate）。一个只有3个字段的类，手写 `__init__`、`__repr__`、`__eq__` 要20多行，dataclass 只需要5行。DRY 原则的体现。
>
> **🔄 举一反三**
> - 配置对象、DTO（数据传输对象）→ 用 dataclass
> - 需要不可变数据？用 `@dataclass(frozen=True)`
> - Kotlin 的 data class、Java 的 record 是类似概念

```python
from dataclasses import dataclass, field

@dataclass
class Point:
    x: float
    y: float

@dataclass(frozen=True)  # 不可变
class Color:
    r: int
    g: int
    b: int

p = Point(3, 4)
print(p)  # Point(x=3, y=4) — 自动生成 __repr__
```

### 11.1 ★ 工程化 dataclass：default_factory / __post_init__ / InitVar

> 实际项目中 90% 的 dataclass 都需要这些特性。

#### default_factory：可变默认值

```python
from dataclasses import dataclass, field

@dataclass
class User:
    name: str
    tags: list[str] = field(default_factory=list)    # ★ 必须用 factory
    # tags: list[str] = []   # ❌ ValueError！dataclass 阻止可变默认值
```

→ dataclass 在 3.11+ 主动拦截 list/dict/set 默认值（防 7.6 默认参数陷阱）。

#### __post_init__：初始化后钩子

```python
@dataclass
class Rectangle:
    width: float
    height: float
    area: float = field(init=False)    # 不进入 __init__ 参数
    
    def __post_init__(self):
        self.area = self.width * self.height

r = Rectangle(3, 4)
print(r.area)    # 12.0
```

#### InitVar：仅初始化用，不存为属性

```python
from dataclasses import dataclass, field, InitVar
import hashlib

@dataclass
class Account:
    username: str
    password: InitVar[str]                  # ← 不会存为 self.password
    password_hash: str = field(init=False)
    
    def __post_init__(self, password: str):  # ← __post_init__ 收到 InitVar
        self.password_hash = hashlib.sha256(password.encode()).hexdigest()

a = Account("alice", "secret123")
print(a.password_hash)
# print(a.password)   # ❌ AttributeError：InitVar 不存为属性
```

#### slots=True（3.10+）：节省内存

```python
@dataclass(slots=True)
class Point:
    x: float
    y: float

# 自动生成 __slots__ = ('x', 'y')
# 内存节省 30-40%，属性访问稍快
# 代价：不能动态加属性
```

#### frozen + hash：用作字典 key

```python
@dataclass(frozen=True)         # frozen 自动让对象 hashable
class Coord:
    x: int
    y: int

d = {Coord(1, 2): "origin"}     # ✅ 可以做 key
# c = Coord(1, 2); c.x = 5      # ❌ FrozenInstanceError
```

> ⚠️ **frozen 不是真正不可变**：仍可改可变属性的内容（如 `frozen` dataclass 内嵌 `list`）。要完全不可变需要 `__slots__` + 全用不可变类型。

## 12. Mixin 与多继承

> **🤔 为什么需要 Mixin？**
> 继承有个问题：单继承不够用（想同时有序列化能力和日志能力），多继承太混乱。Mixin 是一种**轻量级的"能力插件"**——你需要什么能力就"混入"什么。
>
> **🎯 类比理解**
> Mixin = 手机壳上的功能模块。手机本身是核心（主继承链），但你可以加上"支架模块"、"卡包模块"、"充电模块"（Mixin）。每个模块独立提供一种能力。
>
> **💡 设计思路**
> Mixin 的设计原则：
> 1. 不应该被独立实例化
> 2. 只提供一个小功能
> 3. 不应该有 `__init__`（避免初始化冲突）
> 4. 方法名要有辨识度，避免冲突
>
> 这是**组合优于继承**的具体实践——通过混入小模块来组装能力，而不是构建深层继承链。
>
> **🔄 举一反三**
> - Django 的 LoginRequiredMixin, PermissionMixin
> - 任何"XXXMixin"都是一个独立可插拔的能力模块
> - 思考：Mixin vs 装饰器？Mixin 给类加能力，装饰器给函数加能力

```python
class SerializeMixin:
    def to_json(self):
        import json
        return json.dumps(self.__dict__, ensure_ascii=False)

class LogMixin:
    def log(self, msg):
        print(f"[{self.__class__.__name__}] {msg}")

class User(SerializeMixin, LogMixin):
    def __init__(self, name, email):
        self.name = name
        self.email = email

user = User("小明", "xm@test.com")
user.log("创建用户")     # [User] 创建用户
print(user.to_json())    # {"name": "小明", "email": "xm@test.com"}
```

## 总结：面向对象的思维模型

| 思维原则 | 核心问题 | 对应知识点 |
|---------|---------|-----------|
| 建模 | "现实世界里这个东西有什么属性和行为？" | 类定义 |
| 复用 | "这些类有什么共同点可以提取？" | 继承 |
| 统一 | "能用同一个接口处理不同对象吗？" | 多态、鸭子类型 |
| 隐藏 | "调用者需要知道内部细节吗？" | 封装、property |
| 约束 | "子类必须实现哪些方法？" | 抽象类 |
| 组合 | "能用小模块拼装出大功能吗？" | Mixin |
| 协议 | "想让对象支持什么操作？" | 魔术方法 |

**核心思路：OOP 不是为了"用类"而用类，而是为了让代码更接近人类思维方式——用"对象"和"交互"来组织程序逻辑。**

### 练习 1（基础）

实现一个 `Rectangle` 类，包含 `width`、`height`，提供 `area()` 和 `perimeter()` 方法。

<details><summary>答案</summary>

```python
class Rectangle:
    def __init__(self, width: float, height: float):
        self.width = width
        self.height = height
    
    def area(self) -> float:
        return self.width * self.height
    
    def perimeter(self) -> float:
        return 2 * (self.width + self.height)
```

### 练习 3（魔术方法）

实现 `Vector` 类，支持 `+`、`-`、`*`（与标量）、`==` 和 `repr`。

<details><summary>答案</summary>

```python
class Vector:
    def __init__(self, x, y):
        self.x, self.y = x, y
    
    def __add__(self, other):
        return Vector(self.x + other.x, self.y + other.y)
    
    def __sub__(self, other):
        return Vector(self.x - other.x, self.y - other.y)
    
    def __mul__(self, scalar):
        return Vector(self.x * scalar, self.y * scalar)
    
    def __eq__(self, other):
        return isinstance(other, Vector) and self.x == other.x and self.y == other.y
    
    def __repr__(self):
        return f"Vector({self.x}, {self.y})"
```
</details>

### 练习 4（property）

`Celsius` 类，温度不能低于 -273.15。用 property 实现验证。

<details><summary>答案</summary>

```python
class Celsius:
    def __init__(self, temp):
        self.temp = temp
    
    @property
    def temp(self):
        return self._temp
    
    @temp.setter
    def temp(self, value):
        if value < -273.15:
            raise ValueError("低于绝对零度")
        self._temp = value
```
</details>

### 练习 5（单例）

用 `__new__` 实现单例。

<details><summary>答案</summary>

```python
class Singleton:
    _instance = None
    
    def __new__(cls, *args, **kwargs):
        if cls._instance is None:
            cls._instance = super().__new__(cls)
        return cls._instance
```
