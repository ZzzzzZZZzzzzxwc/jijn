## 1. 条件判断

> **🤔 为什么需要条件判断？**
> 程序不能是一条直路——要根据不同情况做不同的事。条件判断就是程序的"岔路口"：红灯停、绿灯行。
>
> **🎯 类比理解**
> if/elif/else = 流程图的菱形判断框。每个菱形问一个问题，根据答案走不同分支。
>
> **💡 设计思路**
> Python 用缩进代替大括号来标识代码块，这迫使你写出结构清晰的代码——缩进就是你的逻辑层次。另外 Python 没有 switch（3.10前），因为 if/elif 足够清晰。
>
> **🔄 举一反三**
> - 卫语句（Guard Clause）：先处理异常情况并 return，避免深层嵌套
> - 短路求值：`x = a or b` 用来设默认值
> - 思考：如果 if 嵌套超过3层，你的代码结构可能有问题——考虑提前返回或拆函数

### if / elif / else

```python
score = 85
if score >= 90:
    grade = "优秀"
elif score >= 80:
    grade = "良好"
elif score >= 60:
    grade = "及格"
else:
    grade = "不及格"
```

### 条件表达式（三元运算符）

```python
status = "成年" if age >= 18 else "未成年"
```

### 逻辑运算与短路求值

> **💡 思路：短路求值不只是优化，还是一种编程技巧**

```python
# or: 第一个为真就返回第一个，否则返回第二个
name = user_input or "匿名"       # 空字符串时用默认值

# and: 第一个为假就返回第一个，否则返回第二个
result = data and data.process()  # data为None时安全，不会报错
```

### 成员与身份判断

> **💡 关键区别：`==` 比较值，`is` 比较身份（是不是同一个对象）**

```python
if "apple" in fruits:    # 成员判断
    pass

if x is None:            # 身份判断（None 用 is）
    pass

a = [1, 2, 3]
b = [1, 2, 3]
a == b   # True（值相同）
a is b   # False（不是同一个对象）
```

### match-case (Python 3.10+)

> **🤔 为什么加了 match-case？**
> if/elif 链在处理"结构化数据匹配"时很笨拙。match-case 不只是 switch——它能**解构数据**，匹配模式，一步到位。
>
> **🎯 类比理解**
> match = 快递分拣机。不只看标签（值匹配），还能拆开看里面是什么形状（结构匹配）。
>
> **🔄 举一反三**
> - 解析命令：match command → "quit" / "help" / ...
> - 处理 API 响应：match response → {"status": 200, "data": d} / {"error": e}
> - Rust/Haskell 的模式匹配是同一概念

```python
point = (3, 4)
match point:
    case (0, 0):
        print("原点")
    case (x, 0):
        print(f"x轴, x={x}")
    case (x, y):
        print(f"点 ({x}, {y})")

# 匹配字典结构
config = {"type": "circle", "radius": 5}
match config:
    case {"type": "circle", "radius": r}:
        print(f"圆, 半径={r}")
    case {"type": "rect", "width": w, "height": h}:
        print(f"矩形 {w}x{h}")
```

## 2. 循环

> **🤔 为什么需要循环？**
> 计算机最擅长的就是"重复做事"。循环让你用几行代码完成上万次操作——这就是编程的杠杆效应。
>
> **🎯 类比理解**
> - for 循环 = 旅游团跟导游走（有明确路线和终点）
> - while 循环 = 钓鱼（不知道什么时候能钓到，满足条件才停）
>
> **💡 设计思路**
> Python 的 for 循环本质是**迭代器协议**——不是"从i=0数到n"，而是"从容器里一个个取元素"。这让它能遍历任何可迭代对象（列表、文件、生成器...），而不仅是数字范围。
>
> **🔄 举一反三**
> - 能用 for 就不用 while（更安全，不会无限循环）
> - 需要索引？用 `enumerate`，别用 `range(len(x))`
> - 同时遍历多个序列？用 `zip`
> - 思考：Python 没有 do-while，如何模拟？（while True + break）

### for 循环

```python
# 遍历任何可迭代对象
for item in [1, 2, 3]: ...
for char in "Python": ...
for key, value in data.items(): ...
for i in range(5): ...           # 0,1,2,3,4
for i in range(0, 20, 3): ...   # 步长3
```

### while 循环

```python
# 条件循环——不知道要循环多少次时用
while not found:
    result = search_next()
    if result:
        found = True

# 无限循环 + break（常见模式）
while True:
    cmd = input(">>> ")
    if cmd == "quit":
        break
```

### break, continue, else

> **💡 思路：for/while 的 else 是"循环没被 break 时执行"——很适合"搜索"场景**

```python
# break: 找到就停
for item in items:
    if item == target:
        print("找到了!")
        break

# continue: 跳过不需要的
for i in range(10):
    if i % 2 == 0:
        continue   # 跳过偶数
    print(i)       # 只打印奇数

# for...else: 没找到时执行 else
for n in range(2, 100):
    for d in range(2, n):
        if n % d == 0:
            break          # 能整除 → 不是质数
    else:
        print(f"{n} 是质数")  # 没被 break → 是质数
```

> **🔄 举一反三**
> for...else 的思维模型："找到就 break，没找到走 else"。这比设一个 `found = False` 标志位更优雅。

### 实用迭代工具

```python
# enumerate: 带索引遍历（别用 range(len(x))!）
for i, fruit in enumerate(["苹果", "香蕉", "橙子"], start=1):
    print(f"{i}. {fruit}")

# zip: 并行遍历
for name, score in zip(names, scores):
    print(f"{name}: {score}")

# reversed: 反向
for x in reversed(range(5)):
    print(x)  # 4,3,2,1,0
```

### 海象运算符 := (Python 3.8+)

> **💡 思路：在表达式中赋值，减少重复计算**

```python
# 传统写法：计算两次或多一个变量
while True:
    line = input(">>> ")
    if line == "quit":
        break

# 海象运算符：赋值+判断一步到位
while (line := input(">>> ")) != "quit":
    print(f"你输入了: {line}")

# 列表推导中避免重复计算
results = [y for x in data if (y := expensive(x)) > threshold]
```

## 3. 异常处理

> **🤔 为什么需要异常处理？**
> 程序运行中意外无处不在：文件不存在、网络断开、用户输入垃圾数据。异常处理让程序能**优雅地应对意外**，而不是直接崩溃。
>
> **🎯 类比理解**
> 异常处理 = 汽车安全气囊。平时不起作用，但一旦出事就能保护你（程序）不"死"。
> try = "尝试做这件事"
> except = "如果出事了怎么办"
> finally = "不管出没出事都要做的善后"
>
> **💡 设计思路**
> Python 采用 **EAFP 风格**（Easier to Ask Forgiveness than Permission）：先做，出错再处理。而不是 LBYL（Look Before You Leap）：先检查所有条件再做。
>
> EAFP 更 Pythonic 的原因：
> 1. 检查条件和执行操作之间可能有时间差（竞态条件）
> 2. 多数情况不会出错，先检查是浪费
> 3. 代码更简洁
>
> **🔄 举一反三**
> - 文件操作：try open 比先 if exists 再 open 更安全（文件可能在检查后被删）
> - 字典取值：try d[key] except KeyError 是 EAFP
> - 网络请求：一定会失败，必须有异常处理

### 基本语法

```python
try:
    result = 10 / 0
except ZeroDivisionError:
    print("除零错误!")
except (TypeError, ValueError) as e:
    print(f"类型/值错误: {e}")
except Exception as e:
    print(f"其他错误: {e}")
else:
    print(f"成功: {result}")    # 没出异常时执行
finally:
    print("清理工作")           # 无论如何都执行
```

> **💡 各部分的思路：**
> - `except`: 具体处理特定错误
> - `else`: 没出错才做的事（比如提交数据库事务）
> - `finally`: 必须做的清理（关文件、释放锁）

### 常见异常层次

```python
# 记住最常用的几个：
# ValueError   — 值不对（int("abc")）
# TypeError    — 类型不对（"1" + 1）
# KeyError     — 字典 key 不存在
# IndexError   — 索引越界
# AttributeError — 属性不存在
# FileNotFoundError — 文件不在
# ZeroDivisionError — 除以零
```

### 抛出与自定义异常

> **💡 思路：什么时候该 raise？当函数遇到它无法处理的情况时，抛给调用者决定**

```python
def divide(a, b):
    if b == 0:
        raise ValueError("除数不能为零")  # 我处理不了，你来决定
    return a / b
```

> **💡 自定义异常的思路：给错误分类，让调用者能精准处理**

```python
class AppError(Exception):
    """应用基础异常"""
    pass

class ValidationError(AppError):
    def __init__(self, field, message):
        self.field = field
        self.message = message
        super().__init__(f"{field}: {message}")

class NotFoundError(AppError):
    def __init__(self, resource, id):
        super().__init__(f"{resource}(id={id}) 不存在")

# 调用者可以分级处理
try:
    create_user("", 20)
except ValidationError as e:
    print(f"输入有误: {e.field}")  # 精准处理
except AppError as e:
    print(f"应用错误: {e}")        # 兜底
```

### 最佳实践

```python
# ✅ 捕获具体异常
try:
    value = config["key"]
except KeyError:
    value = "default"

# ❌ 不要用裸 except（会吞掉所有错误，包括 bug）
# try: ...
# except: pass

# ✅ EAFP 风格
try:
    return data[key]
except KeyError:
    return default

# ✅ 忽略特定异常的优雅写法
from contextlib import suppress
with suppress(FileNotFoundError):
    os.remove("temp.txt")
```

### 3.4b ★ 生产环境异常日志最佳实践

> **💡 工程原则：异常必须记录完整 traceback，否则线上排障如盲人摸象。**

```python
import logging

logger = logging.getLogger(__name__)

# ✅ 正确：logging.exception() 自动记录 traceback
try:
    result = process_order(order_id)
except ValidationError as e:
    logger.warning("订单验证失败: %s", e)  # 业务异常，warning 够了
except Exception as e:
    logger.exception("处理订单 %s 时发生未知错误", order_id)
    # exception() = error() + 自动附加 traceback
    raise  # 未知错误应该 re-raise

# ❌ 错误：丢失了 traceback
# except Exception as e:
#     logger.error(f"出错了: {e}")   # 只有错误消息，没有调用栈！
```

**异常处理决策树**：
```
捕获异常后该怎么办？
├── 能恢复？→ 处理并继续（如用默认值、重试）
├── 不能恢复但需要转换？→ raise NewError from e（异常链）
├── 不能恢复？→ logger.exception() + raise
└── 确实可以忽略？→ contextlib.suppress()（极少数情况）
```

### 3.5 ★ 异常链：raise X from Y

> 生产代码必备。**没有异常链 = 调试地狱**。

### 显式链接（`from`）

```python
def load_config(path):
    try:
        return open(path).read()
    except FileNotFoundError as e:
        raise ConfigError(f"配置文件丢失: {path}") from e
        # ↑ from e 把原始异常链接到 __cause__
```

输出：
```
FileNotFoundError: [Errno 2] No such file or directory: 'cfg.json'

The above exception was the direct cause of the following exception:

ConfigError: 配置文件丢失: cfg.json
```

### 三种链接语义

| 写法 | `__cause__` | `__context__` | traceback 显示 |
|---|---|---|---|
| `raise X from Y` | Y（**显式因果**） | Y | "**direct cause**" |
| `except: raise X` | None | 自动 = 当前异常 | "during handling of" |
| `raise X from None` | None | 仍设置 | **隐藏链**（仅显示 X） |

```python
# 隐藏旧异常（API 边界常用）
def fetch_user(uid):
    try:
        return db.query(uid)
    except DatabaseError:
        raise UserNotFound(uid) from None    # 调用者不需要知道是 DB 问题
```

### 自动链接（隐式 context）

```python
try:
    1/0
except ZeroDivisionError:
    raise ValueError("oops")    # 没用 from，但仍链接

# __context__ = ZeroDivisionError
# __cause__ = None
# __suppress_context__ = False  → 仍打印两个 traceback
```

### ⚠️ except 块结束后异常变量被自动 del

```python
try:
    raise ValueError("x")
except ValueError as e:
    err = e        # 想保留？必须显式赋给别的名字
    
# 这里 e 已被自动 del（防止循环引用：e → traceback → frame → e）
print(e)           # ❌ NameError
print(err)         # ✅
```

### 3.6 ★ ExceptionGroup 与 except*（3.11+）

> **TaskGroup 的基础**——asyncio 多任务并发时收集多个错误。

### 创建与抛出

```python
errors = [
    ValueError("a"),
    KeyError("b"),
    TypeError("c"),
]
raise ExceptionGroup("批处理失败", errors)
```

### except* 分组捕获

```python
try:
    raise ExceptionGroup("multi", [
        ValueError("v"),
        KeyError("k"),
        TypeError("t"),
    ])
except* ValueError as eg:
    print(f"值错误: {[str(e) for e in eg.exceptions]}")
except* (KeyError, TypeError) as eg:
    print(f"其他: {[type(e).__name__ for e in eg.exceptions]}")
# 输出：
# 值错误: ['v']
# 其他: ['KeyError', 'TypeError']
```

→ 与传统 `except` 不同：**所有匹配的分支都执行**（异常被"分流"）。

### 实际场景：asyncio.TaskGroup

```python
import asyncio

async def main():
    async with asyncio.TaskGroup() as tg:
        tg.create_task(fetch_a())
        tg.create_task(fetch_b())
        tg.create_task(fetch_c())
# 任一 task 抛错 → TaskGroup 取消其他 task → 收集为 ExceptionGroup 抛出

try:
    asyncio.run(main())
except* ConnectionError as eg:
    log_network_errors(eg.exceptions)
except* ValueError as eg:
    log_validation_errors(eg.exceptions)
```

→ 没有 ExceptionGroup 之前，多个并发错误只能"丢一个、保一个"，3.11 终于解决。

## 4. 上下文管理器

> **🤔 为什么需要上下文管理器？**
> 很多操作需要"成对执行"：打开文件→关闭文件，获取锁→释放锁，连接数据库→断开连接。如果忘了"关"的那一步，就会资源泄漏。上下文管理器保证**不管出不出错，清理工作一定执行**。
>
> **🎯 类比理解**
> with 语句 = "借/还"模式。图书馆借书：进入 with 块 = 借书（`__enter__`），离开 with 块 = 还书（`__exit__`）。不管你有没有看完书，到期一定要还。
>
> **💡 设计思路**
> 上下文管理器把"资源管理"从业务逻辑中分离出来。你只需要在 with 块里写核心逻辑，资源的获取和释放由 with 自动保证。这消灭了"忘记关文件/释放锁"这类 bug。
>
> **🔄 举一反三**
> - 文件操作、数据库连接、网络请求、线程锁 → 都该用 with
> - 测试中的临时环境（mock、临时文件）→ with
> - 思考：没有 with 时你得 try/finally 手动清理，多丑多容易出错

### with 语句

```python
# 最常见：文件操作
with open("file.txt", "r") as f:
    content = f.read()
# 离开 with 块后，文件保证被关闭（即使中间报错）

# 多个上下文管理器
with open("in.txt") as fin, open("out.txt", "w") as fout:
    fout.write(fin.read())
```

### 自己实现上下文管理器

> **💡 两种方式：类（实现协议）或 @contextmanager（更简洁）**

```python
# 方式1：类
class Timer:
    def __enter__(self):
        import time
        self.start = time.perf_counter()
        return self
    
    def __exit__(self, exc_type, exc_val, exc_tb):
        import time
        self.elapsed = time.perf_counter() - self.start
        print(f"耗时: {self.elapsed:.4f}s")
        return False  # False=不抑制异常

with Timer():
    sum(range(1000000))

# 方式2：@contextmanager（推荐，更简洁）
from contextlib import contextmanager

@contextmanager
def timer(label=""):
    import time
    start = time.perf_counter()
    try:
        yield  # yield 前 = __enter__, yield 后 = __exit__
    finally:
        elapsed = time.perf_counter() - start
        print(f"{label} 耗时: {elapsed:.4f}s")

with timer("计算"):
    sum(range(1000000))
```

> **💡 @contextmanager 的思维模型：**
> ```
> yield 之前的代码 = 进入时执行（__enter__）
> yield 的值 = as 后面的变量
> yield 之后的代码 = 退出时执行（__exit__）
> finally 保证清理一定执行
> ```

### 实用场景

```python
# 数据库事务
@contextmanager
def transaction(conn):
    try:
        yield conn
        conn.commit()    # 没出错就提交
    except:
        conn.rollback()  # 出错就回滚
        raise

# 临时修改状态
@contextmanager
def temporary_setting(obj, attr, value):
    old = getattr(obj, attr)
    setattr(obj, attr, value)
    try:
        yield
    finally:
        setattr(obj, attr, old)  # 恢复原值
```

## 5. 断言 assert

> **🤔 为什么需要断言？**
> 断言是给**开发者自己**用的——"我断定这里一定满足某条件，如果不满足说明我的代码有 bug"。它是开发阶段的安全网。
>
> **🎯 类比理解**
> assert = 脚手架上的安全绳。施工时（开发阶段）保护你，完工后（生产环境）可以拆掉（-O 模式下 assert 被忽略）。
>
> **💡 设计思路**
> assert 与异常处理的区别：
> - **assert** = "这不应该发生，如果发生了是我的代码有 bug"（内部检查）
> - **raise** = "这可能发生，调用者传了错误的输入"（外部输入验证）
>
> 永远不要用 assert 做输入验证！因为 `python -O` 会跳过所有 assert。
>
> **🔄 举一反三**
> - 函数入口检查前置条件：`assert len(items) > 0`
> - 算法中间验证不变量：`assert total == sum(parts)`
> - 测试中验证结果：断言是单元测试的核心

```python
def calculate_avg(numbers):
    assert len(numbers) > 0, "列表不能为空"  # 开发时的安全网
    return sum(numbers) / len(numbers)

# ❌ 不要这样做（生产环境 assert 会被跳过）
# assert user_input.isdigit(), "必须输入数字"

# ✅ 应该用 raise
if not user_input.isdigit():
    raise ValueError("必须输入数字")
```

---

## 总结：控制流的思维模型

| 思维原则 | 核心问题 | 对应知识点 |
|---------|---------|-----------|
| 分支 | "不同情况做不同的事" | if/elif/else, match |
| 重复 | "同样的事做很多次" | for, while |
| 提前退出 | "异常情况先处理掉" | 卫语句, break, continue |
| 容错 | "出错了怎么办？" | try/except |
| 资源安全 | "借了一定要还" | with, 上下文管理器 |
| 自检 | "我的代码逻辑对吗？" | assert |

**核心思路三条：**

1. **减少嵌套**：用卫语句提前返回、用 continue 跳过、用函数提取 → 让代码扁平化
2. **EAFP > LBYL**：先做再说比先检查再做更 Pythonic（也更安全）
3. **资源用 with 管理**：任何"获取→使用→释放"模式都该用上下文管理器

def parse_int(s: str) -> int | None:
    try:

    except (ValueError, TypeError):

class ValidationError(Exception):
    def __init__(self, field: str, message: str):

### 练习 3（with）

实现一个 `Timer` 上下文管理器，打印代码块耗时。

<details><summary>答案</summary>

```python
import time

class Timer:
    def __enter__(self):
        self.start = time.perf_counter()
        return self
    
    def __exit__(self, *args):
        elapsed = time.perf_counter() - self.start
        print(f"耗时 {elapsed:.4f}s")

with Timer():
    sum(range(10**6))
```

或用 `@contextmanager`：

```python
from contextlib import contextmanager

@contextmanager
def timer():
    start = time.perf_counter()
    try:
        yield
    finally:
        print(f"耗时 {time.perf_counter() - start:.4f}s")
```
</details>
