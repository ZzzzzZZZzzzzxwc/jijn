## 15.1 项目结构

### 推荐结构

```
my_project/
├── .gitignore
├── README.md
├── pyproject.toml          # ★ 现代标准（取代 setup.py / requirements.txt）
├── .python-version         # 指定 Python 版本（pyenv）
├── src/
│   └── my_project/
│       ├── __init__.py
│       ├── __main__.py     # 支持 python -m my_project
│       ├── cli.py
│       ├── core/
│       ├── api/
│       └── utils/
├── tests/
│   ├── __init__.py
│   ├── conftest.py
│   ├── test_core.py
│   └── test_api.py
├── docs/
└── scripts/
```

**为什么用 src/ 布局**：
- 强制通过安装才能 import（避免"在项目根目录就能 import"导致的导入路径混乱）
- 测试不会意外 import 到本地源码而非已安装版本

## 15.3 虚拟环境与依赖管理

### 15.3.1 venv（标准库）

```bash
python -m venv .venv
source .venv/bin/activate         # Linux/macOS
.venv\Scripts\activate            # Windows

pip install -e ".[dev]"           # 安装本项目（可编辑模式）+ dev 依赖
```

### 15.3.2 推荐工具：uv 或 poetry

**uv**（Rust 实现，极快）：
```bash
pip install uv
uv venv
uv pip install -e ".[dev]"
uv pip compile pyproject.toml -o requirements.txt
```

**poetry**：
```bash
poetry init
poetry add requests
poetry add --group dev pytest
poetry install
```

### 15.3.3 ★ 锁定版本

```bash
pip freeze > requirements.lock      # 简单粗暴
# 或用 pip-tools
pip-compile                         # 生成 requirements.txt 锁定版本
```

部署时用 `pip install -r requirements.lock`，保证环境一致。

## 15.5 日志

### 15.5.1 ⚠️ 不要用 print 调试生产代码

```python
# ❌
print(f"user {uid} logged in")

# ✅
import logging
logger = logging.getLogger(__name__)
logger.info("user %s logged in", uid)
```

### 15.5.2 标准配置

```python
import logging
import logging.config

LOGGING_CONFIG = {
    "version": 1,
    "disable_existing_loggers": False,
    "formatters": {
        "default": {
            "format": "%(asctime)s [%(levelname)s] %(name)s: %(message)s"
        },
    },
    "handlers": {
        "console": {
            "class": "logging.StreamHandler",
            "formatter": "default",
            "level": "INFO",
        },
        "file": {
            "class": "logging.handlers.RotatingFileHandler",
            "filename": "app.log",
            "maxBytes": 10*1024*1024,
            "backupCount": 5,
            "formatter": "default",
            "level": "DEBUG",
        },
    },
    "root": {
        "handlers": ["console", "file"],
        "level": "INFO",
    },
}

logging.config.dictConfig(LOGGING_CONFIG)
```

### 15.5.3 结构化日志（推荐）

```python
import structlog

logger = structlog.get_logger()
logger.info("login", user_id=42, ip="1.2.3.4")
# {"event": "login", "user_id": 42, "ip": "1.2.3.4", "timestamp": "..."}
```

便于日志聚合系统（ELK、Loki）查询。

### 15.5.4 日志级别

| 级别 | 用途 |
|------|------|
| DEBUG | 详细调试信息 |
| INFO | 正常运行节点 |
| WARNING | 潜在问题，不影响运行 |
| ERROR | 错误，但程序还能继续 |
| CRITICAL | 致命错误 |

## 15.8 错误处理与可靠性

### 15.8.1 不要静默吞错

### 15.8.2 使用自定义异常

```python

    """所有业务异常的基类"""

    pass
```

### 15.8.3 重试与超时

```python
from tenacity import retry, stop_after_attempt, wait_exponential

@retry(stop=stop_after_attempt(3),
       wait=wait_exponential(multiplier=1, min=1, max=10))

    return requests.get(url, timeout=10)
```

```python
# ✅ 网络请求设计为幂等
```

## 15.10 部署

### 15.10.1 Dockerfile（多阶段构建）

```dockerfile
# === 构建阶段 ===
FROM python:3.12-slim AS builder
WORKDIR /app

# ★ 必须先 COPY 源码，否则 pip install . 会找不到包
COPY pyproject.toml README.md ./
COPY src/ ./src/

# ★ 不要用 -e（editable）：生产镜像里 editable 让"改容器内源码即生效"，违反不可变镜像原则
RUN pip install --user --no-cache-dir .

# === 运行阶段 ===
FROM python:3.12-slim
WORKDIR /app
COPY --from=builder /root/.local /root/.local
ENV PATH=/root/.local/bin:$PATH \
    PYTHONDONTWRITEBYTECODE=1 \
    PYTHONUNBUFFERED=1

# ★ 不要用 root 跑应用
RUN useradd -u 1000 -m app
USER app

CMD ["python", "-m", "my_project"]
```

> ⚠️ **常见 Dockerfile 陷阱**：
> 1. **只 COPY pyproject.toml 然后 `pip install .`** → 找不到包源码，构建失败
> 2. **生产镜像用 `pip install -e .`** → editable 模式让源码可变，破坏镜像不可变性
> 3. **以 root 跑应用** → 容器逃逸时风险放大
> 4. **不设 `PYTHONUNBUFFERED=1`** → 容器日志被 Python 行缓冲卡住，看不到实时输出

### 15.13.4 SQL 注入

```python
# ❌ 字符串拼接——攻击者传 name = "' OR 1=1 --" 即可绕过
cursor.execute(f"SELECT * FROM users WHERE name = '{name}'")

# ✅ 参数化查询（占位符随驱动不同！）
# psycopg / PyMySQL / mysql-connector  → %s
cursor.execute("SELECT * FROM users WHERE name = %s", (name,))

# sqlite3                              → ?
cursor.execute("SELECT * FROM users WHERE name = ?", (name,))

# SQLAlchemy Core                      → :name
from sqlalchemy import text
session.execute(text("SELECT * FROM users WHERE name = :name"), {"name": name})
```

> 🔬 **DB-API 2.0 没有统一占位符**——`paramstyle` 由驱动决定（`format` / `qmark` / `named` / `numeric`）。  
> 检查方法：`import psycopg2; psycopg2.paramstyle  # 'pyformat'`。  
> ⚠️ **永远不要相信"我用 f-string 但只传内部值"**——一次重构把内部值改成用户输入就完蛋。

## 15.13.5 ★ 反序列化安全：pickle / yaml / shelve

> ⚠️ **`pickle.loads()` 可以执行任意代码**——这是 Python 安全的头号陷阱。

```python
import pickle

# ❌ 攻击演示（不要在生产环境跑！）
# 构造一个恶意 pickle 数据
malicious = b"cos\nsystem\n(S'echo PWNED'\ntR."
# pickle.loads(malicious)  # 会执行 os.system('echo PWNED')!
```

**原理**：pickle 协议支持 `__reduce__` 方法，允许指定"反序列化时调用的任意可调用对象+参数"。攻击者可以构造调用 `os.system`、`subprocess.Popen` 等的 payload。

**高危场景**：
- 从 Redis / 消息队列收到 pickle 数据
- 用户上传的文件用 `pickle.load` 解析
- `shelve` 底层就是 pickle
- `yaml.load(data)` 默认不安全（`!!python/object` 标签可 RCE）

**铁律**：

| 操作 | 安全 | 替代 |
|------|------|------|
| `pickle.loads(untrusted)` | ❌ **永远不做** | `json.loads` / `msgpack` / `orjson` |
| `yaml.load(data)` | ❌ 允许 RCE | `yaml.safe_load(data)` |
| `shelve.open(path)` | ❌ 如果 path 不可信 | sqlite3 / json 文件 |
| `pickle.loads(trusted_internal)` | ✅ 仅限内部数据 | — |

```python
# ✅ 安全替代
import json, yaml

data = json.loads(user_input)           # 只能解析 JSON 值，无代码执行风险
config = yaml.safe_load(config_file)    # 只允许标量/序列/映射，禁止 !!python

# 高性能序列化（替代 pickle）
import orjson  # 比 json 快 3-10x，不允许代码执行
data = orjson.loads(payload)
```

## 15.13.6 logger.propagate 与重复日志（生产头号 logging 故障）

```python
import logging

# 典型错误：给子 logger 加了 handler，但 propagate=True（默认）
app_logger = logging.getLogger("myapp")
app_logger.addHandler(logging.StreamHandler())   # handler A
app_logger.setLevel(logging.INFO)

# root logger 也有 handler（logging.basicConfig 自动加的）
logging.basicConfig()                             # handler B（root）

app_logger.info("hello")
# 输出两遍 "hello"！
# 原因：消息先被 handler A 处理，然后 propagate 到 root logger 被 handler B 再处理一遍
```

**修复**：

```python
# 方案 1：关闭 propagate
app_logger.propagate = False

# 方案 2：只在 root 配 handler，子 logger 不加（推荐）
logging.basicConfig(level=logging.INFO, format="%(name)s: %(message)s")
logger = logging.getLogger("myapp")   # 不加 handler，利用 propagate 到 root
logger.info("hello")                   # 只输出一遍
```

> 🔬 **propagate 链路**：logger.info → 自己的 handlers → 如果 propagate=True → 父 logger 的 handlers → … → root。**每一级只要有 handler 就会输出一遍**。

## 15.15 面试常问

### Q1：怎么管理 Python 项目依赖？

**答**：用 `pyproject.toml` 声明依赖，用 `uv`/`poetry`/`pip-compile` 锁定版本。部署时用锁定版本保证一致性。

### Q2：单元测试和集成测试的区别？

**答**：
- 单元测试：测单一函数/类，外部依赖用 mock
- 集成测试：测多个组件协作，可能用真实数据库
- 单元测试快、独立；集成测试慢但更接近真实

### Q3：怎么处理生产环境的配置？

**答**：通过环境变量注入，不写在代码里。本地开发用 `.env`（不提交）。可用 pydantic-settings 自动加载和验证。

### Q4：日志级别怎么选？

**答**：DEBUG 给开发看，INFO 记录关键节点，WARNING 是潜在问题，ERROR 是真错误，CRITICAL 是致命问题。生产一般 INFO+。
