## 1. 装饰器基本概念

> **🤔 为什么需要装饰器？**
> 你有10个函数都需要加"日志记录"功能。难道要在每个函数里都复制粘贴日志代码？装饰器让你**不修改原函数代码**就能给它"穿上新衣服"——添加新功能。
>
> **🎯 类比理解**
> 装饰器 = 手机壳。手机（原函数）功能不变，但套上壳（装饰器）后有了新能力——防摔、支架、卡包。而且换壳不需要拆手机。
> 也像三明治：面包是装饰器的前后逻辑，中间的肉是原函数。
>
> **💡 设计思路**
> 装饰器的本质是**高阶函数 + 闭包**的语法糖。核心原则：**开闭原则**——对扩展开放（能加功能），对修改关闭（不改原代码）。这也是 AOP（面向切面编程）思想的体现。
>
> **🔄 举一反三**
> - Java 的注解 `@Override`、Spring 的 `@Transactional` 是类似概念
> - 中间件（Express/Django）本质也是"装饰"请求处理流程
> - 思考：为什么日志、权限、缓存、计时这些"横切关注点"特别适合装饰器？因为它们跟业务逻辑无关，却到处都要用

```python
# 装饰器的本质就是这一行
# say_hello = my_decorator(say_hello)

# @ 语法糖让它更优雅
@my_decorator
def say_hello():
    print("Hello!")
```

### 1.1 执行流程理解

> **💡 关键思路：装饰发生在定义时，执行发生在调用时**

```python
def decorator(func):
    print("装饰阶段")       # ← 导入模块时就执行了
    def wrapper(*args, **kwargs):
        print("调用前")     # ← 调用函数时执行
        result = func(*args, **kwargs)
        print("调用后")     # ← 调用函数时执行
        return result
    return wrapper

@decorator
def greet(name):
    print(f"Hello, {name}!")
# 输出: 装饰阶段（还没调用 greet 就已经打印了！）

greet("小明")
# 调用前 → Hello, 小明! → 调用后
```

### 1.2 🔬 字节码视角：@decorator 到底做了什么

```python
import dis

@decorator
def greet(name):
    print(f"Hello, {name}!")

# 等价于：
# def greet(name): ...
# greet = decorator(greet)

# 用 dis 看编译后的模块级字节码：
```

```
  # 1. 先把 decorator 函数加载到栈上
  LOAD_GLOBAL          decorator         ← 加载装饰器函数

  # 2. 定义 greet 函数（创建 code 对象 + MAKE_FUNCTION）
  LOAD_CONST           <code of greet>
  MAKE_FUNCTION        0

  # 3. 调用 decorator(greet)，结果存回 greet 这个名字
  CALL                 1                  ← decorator(greet)
  STORE_NAME           greet              ← greet = 返回值
```

**关键认知**：
- `@decorator` 是**编译期**的语法糖——编译器生成的字节码里，`MAKE_FUNCTION` 后紧跟 `CALL`
- 装饰发生在**模块加载时**（import 时），不是函数调用时
- 叠加装饰器 `@A @B` → 先 `CALL B`，再 `CALL A`（从内到外）

```python
# 验证：查看装饰后 greet 的真实身份
@decorator
def greet(name): ...

print(greet.__name__)       # 如果没 @wraps → 'wrapper'
print(type(greet))          # <class 'function'>（wrapper 函数）
print(greet.__wrapped__)    # 如果用了 @wraps → 原始 greet
```

> **💡 这就是为什么装饰器能"替换"函数——编译器让 `greet` 这个名字指向了 decorator 的返回值。**

## 2. 函数装饰器

> **🤔 为什么 wrapper 需要 `*args, **kwargs`？**
> 因为你不知道被装饰的函数接受什么参数。`*args, **kwargs` 是"万能接收器"——不管原函数要什么参数，都能透传过去。
>
> **💡 设计思路：装饰器的标准骨架**
> ```
> def decorator(func):      # 接收原函数
>     def wrapper(*args, **kwargs):  # 替代原函数
>         # 前置逻辑
>         result = func(*args, **kwargs)  # 调用原函数
>         # 后置逻辑
>         return result     # 别忘了返回！
>     return wrapper        # 返回新函数
> ```
> 记住这个骨架，所有装饰器都是它的变体。

### 计时装饰器

```python
import time

def timer(func):
    def wrapper(*args, **kwargs):
        start = time.perf_counter()
        result = func(*args, **kwargs)
        elapsed = time.perf_counter() - start
        print(f"{func.__name__} 耗时: {elapsed:.4f}秒")
        return result
    return wrapper

@timer
def slow_function():
    time.sleep(1)
    return "完成"
```

### 日志装饰器

```python
def log_call(func):
    def wrapper(*args, **kwargs):
        print(f"调用 {func.__name__}({args}, {kwargs})")
        result = func(*args, **kwargs)
        print(f"返回 {result!r}")
        return result
    return wrapper
```

> **🔄 举一反三**
> - 计时、日志、重试、缓存——这些"前后包裹"的模式都是装饰器的天然场景
> - 思考：如果把 `result = func(...)` 去掉会怎样？→ 原函数根本不会执行！

## 3. 带参数的装饰器

> **🤔 为什么需要给装饰器传参数？**
> 不同场景需要不同配置。比如重试装饰器——有时要重试3次，有时5次。参数让同一个装饰器适应多种场景。
>
> **🎯 类比理解**
> 无参装饰器 = 固定款手机壳
> 带参装饰器 = 可定制手机壳（选颜色、选材质、选功能）
>
> **💡 设计思路：三层嵌套的逻辑**
> ```
> def decorator_factory(config):  # 第1层：接收配置参数
>     def decorator(func):        # 第2层：接收被装饰函数
>         def wrapper(*args, **kwargs):  # 第3层：替代原函数
>             # 使用 config 和 func
>             pass
>         return wrapper
>     return decorator
> ```
> 为什么要三层？因为每一层负责接收不同的东西：配置→函数→调用参数。
>
> **🔄 举一反三**
> - `@retry(max_attempts=5)` — 重试次数可配置
> - `@cache(ttl=60)` — 缓存时间可配置
> - `@route("/api/users")` — Flask 路由就是带参装饰器

```python
def repeat(times):
    """第1层：接收配置"""
    def decorator(func):
        """第2层：接收函数"""
        def wrapper(*args, **kwargs):
            """第3层：替代函数"""
            for _ in range(times):
                result = func(*args, **kwargs)
            return result
        return wrapper
    return decorator

@repeat(times=3)  # repeat(3) 返回 decorator，decorator(say) 返回 wrapper
def say(msg):
    print(msg)
```

### 兼容有括号和无括号的写法

> **💡 思路：怎么让 `@log` 和 `@log(level="DEBUG")` 都能用？**
> 判断第一个参数是函数还是配置。如果是函数 → 无括号调用；如果不是 → 有括号调用。

```python
def log(func=None, *, level="INFO"):
    def decorator(fn):
        def wrapper(*args, **kwargs):
            print(f"[{level}] {fn.__name__}")
            return fn(*args, **kwargs)
        return wrapper
    
    if func is not None:  # @log（无括号）
        return decorator(func)
    return decorator       # @log(level="DEBUG")（有括号）
```

## 4. 类装饰器

> **🤔 为什么用类来装饰/被装饰？**
> 两种用法：
> 1. **装饰类的函数**：给整个类加功能（如单例、注册）
> 2. **用类作为装饰器**：当装饰器需要维护状态时（如计数），类比闭包更直观
>
> **🎯 类比理解**
> - 装饰类的函数 = 给整栋楼加物业服务（类级别的增强）
> - 用类当装饰器 = 把装饰器做成"有记忆的机器人"（__call__ 就是执行按钮）
>
> **💡 设计思路**
> 当装饰器需要复杂状态（多个变量、需要 reset）时，用类比闭包更清晰。类的 `__init__` 存配置，`__call__` 做执行。

### 单例模式（装饰类的函数）

```python
def singleton(cls):
    instances = {}
    def get_instance(*args, **kwargs):
        if cls not in instances:
            instances[cls] = cls(*args, **kwargs)
        return instances[cls]
    return get_instance

@singleton
class Database:
    def __init__(self, url):
        self.url = url

db1 = Database("localhost")
db2 = Database("localhost")
print(db1 is db2)  # True — 永远只有一个实例
```

### 注册模式

> **💡 思路：装饰器不一定要修改行为，也可以只"登记"**

```python
registry = {}

def register(name):
    def decorator(cls):
        registry[name] = cls  # 登记到注册表
        return cls            # 原封不动返回类
    return decorator

@register("json")
class JsonParser:
    def parse(self, data): ...

@register("csv")
class CsvParser:
    def parse(self, data): ...

# 工厂模式：按名字取出对应的类
parser = registry["json"]()
```

> **🔄 举一反三**
> - Flask 的 `@app.route()` 就是注册模式——登记 URL 和处理函数的对应关系
> - pytest 的 `@pytest.fixture` 也是注册
> - 插件系统的核心就是"注册+查找"

### 用类作为装饰器（有状态）

```python
class CountCalls:
    def __init__(self, func):
        self.func = func
        self.count = 0
    
    def __call__(self, *args, **kwargs):
        self.count += 1
        print(f"{self.func.__name__} 第{self.count}次调用")
        return self.func(*args, **kwargs)

@CountCalls
def hello():
    print("Hello!")

hello()  # hello 第1次调用
hello()  # hello 第2次调用
print(hello.count)  # 2 — 状态保存在实例里
```

## 5. 装饰器叠加

> **🤔 为什么要叠加装饰器？**
> 单一职责：每个装饰器只做一件事。想同时加日志+计时+权限？叠加起来就行。
>
> **🎯 类比理解**
> 叠加装饰器 = 套娃 / 洋葱。最外层先进先出，最内层最贴近原函数。
>
> **💡 核心思路：装饰从下往上，执行从上往下**
> ```python
> @A      # 第二步：A 装饰 B(func)
> @B      # 第一步：B 装饰 func
> def func(): ...
> # 等价于 func = A(B(func))
> # 调用时：A的前置 → B的前置 → func → B的后置 → A的后置
> ```

```python
@bold       # 外层
@italic     # 内层
def greet(name):
    return f"Hello, {name}"

# greet = bold(italic(greet))
print(greet("World"))  # <b><i>Hello, World</i></b>
```

> **🔄 举一反三**
> - 中间件链（Express/Django）就是叠加的装饰器模式
> - 思考顺序时画洋葱图：从外到内进入，从内到外返回

## 6. 保留函数元信息 functools.wraps

> **🤔 为什么需要 @wraps？**
> 装饰后，函数的 `__name__`、`__doc__` 都变成了 wrapper 的。调试时你看到的是 "wrapper" 而不是原函数名——这会让你抓狂。
>
> **🎯 类比理解**
> 没有 @wraps = 给手机套了壳后，手机壳上印的是壳的品牌信息
> 有 @wraps = 手机壳是透明的，还能看到手机原来的品牌和型号
>
> **💡 设计思路**
> @wraps 把原函数的元信息"复制"到 wrapper 上。这是装饰器的"礼貌"——装饰了别人，但不要抹去别人的身份。

```python
from functools import wraps

def my_decorator(func):
    @wraps(func)  # ← 加这一行！
    def wrapper(*args, **kwargs):
        return func(*args, **kwargs)
    return wrapper

@my_decorator
def greet(name):
    """打招呼"""
    return f"Hello, {name}"

print(greet.__name__)  # "greet" ✅ (不加wraps会是"wrapper")
print(greet.__doc__)   # "打招呼" ✅
```

> **🔄 标准模板（背下来）**
> ```python
> from functools import wraps
> def decorator(func):
>     @wraps(func)
>     def wrapper(*args, **kwargs):
>         # 你的逻辑
>         return func(*args, **kwargs)
>     return wrapper
> ```

## 7. 常用内置装饰器

> **💡 设计思路：Python 把最常用的模式内置成装饰器，减少重复造轮子**

### @property — 属性访问控制

```python
class Circle:
    def __init__(self, r):
        self._r = r
    
    @property
    def area(self):  # 像属性一样访问，但背后是计算
        return 3.14 * self._r ** 2
```

### @staticmethod / @classmethod — 方法分类

```python
class MyClass:
    @classmethod
    def create(cls): return cls()    # 替代构造器
    
    @staticmethod
    def utility(): return "工具"     # 纯工具函数
```

### @lru_cache — 自动缓存

> **💡 思路：用空间换时间。相同输入直接返回缓存结果，不再重复计算。**

```python
from functools import lru_cache

@lru_cache(maxsize=128)
def fibonacci(n):
    if n < 2: return n
    return fibonacci(n-1) + fibonacci(n-2)

fibonacci(100)  # 秒出结果（无缓存会算到天荒地老）
```

### @cached_property — 只算一次的属性

```python
from functools import cached_property

class DataAnalysis:
    def __init__(self, data):
        self.data = data
    
    @cached_property
    def stats(self):  # 第一次访问时计算，之后缓存
        return {"mean": sum(self.data)/len(self.data)}
```

> **🔄 举一反三**
> - @property vs @cached_property：每次算 vs 只算一次
> - @lru_cache 适合纯函数（同输入同输出）
> - @total_ordering：只写 `__eq__` + `__lt__`，自动生成其他比较方法

## 8. 实战装饰器模式

> **💡 核心思路：装饰器最适合"横切关注点"——那些跟业务无关但到处要用的东西**

### 权限检查

> **思路：把"谁能用"和"做什么"分开。业务函数只管"做什么"，权限由装饰器负责。**

```python
from functools import wraps

def require_auth(role="user"):
    def decorator(func):
        @wraps(func)
        def wrapper(user, *args, **kwargs):
            if not user.get("authenticated"):
                raise PermissionError("请先登录")
            if user.get("role") != role:
                raise PermissionError(f"需要 {role} 权限")
            return func(user, *args, **kwargs)
        return wrapper
    return decorator

@require_auth(role="admin")
def delete_user(user, user_id):
    return f"已删除用户 {user_id}"
```

### 缓存/记忆化

> **思路：相同的输入没必要算两遍。第一次算完存起来，下次直接查表。**

```python
def memoize(func):
    cache = {}
    @wraps(func)
    def wrapper(*args):
        if args not in cache:
            cache[args] = func(*args)
        return cache[args]
    wrapper.cache = cache
    return wrapper
```

### 输入验证

> **思路：把验证逻辑从业务函数中剥离出来，保持业务函数的纯净。**

```python
def validate_types(**type_hints):
    def decorator(func):
        @wraps(func)
        def wrapper(*args, **kwargs):
            import inspect
            params = list(inspect.signature(func).parameters.keys())
            for i, arg in enumerate(args):
                if params[i] in type_hints:
                    expected = type_hints[params[i]]
                    if not isinstance(arg, expected):
                        raise TypeError(f"{params[i]} 应为 {expected.__name__}")
            return func(*args, **kwargs)
        return wrapper
    return decorator

@validate_types(name=str, age=int)
def create_user(name, age):
    return {"name": name, "age": age}
```

### 限流

> **思路：保护系统不被过度调用。记录调用时间，超频则拒绝。**

```python
import time
from functools import wraps

def rate_limit(max_calls, period):
    calls = []
    def decorator(func):
        @wraps(func)
        def wrapper(*args, **kwargs):
            now = time.time()
            calls[:] = [t for t in calls if t > now - period]
            if len(calls) >= max_calls:
                raise RuntimeError("调用过于频繁")
            calls.append(now)
            return func(*args, **kwargs)
        return wrapper
    return decorator

@rate_limit(max_calls=3, period=10)
def api_call():
    return "响应"
```

> **🔄 举一反三**
> - 所有这些模式的共同点：**把"横切逻辑"和"核心逻辑"分离**
> - 权限、日志、缓存、限流、重试、验证——这些就是典型的"切面"
> - 思考：如果没有装饰器，这些逻辑要嵌入每个函数内部，代码会变成什么样？

## 9. 装饰器高级技巧

### 带状态的装饰器

> **💡 思路：在 wrapper 函数上挂属性来存状态**

```python
def count_calls(func):
    @wraps(func)
    def wrapper(*args, **kwargs):
        wrapper.calls += 1
        return func(*args, **kwargs)
    wrapper.calls = 0  # 状态挂在函数对象上
    return wrapper
```

### 可开关的装饰器

> **💡 思路：通过环境变量/配置控制装饰器是否生效**

```python
import os
def debug_only(func):
    @wraps(func)
    def wrapper(*args, **kwargs):
        if os.environ.get("DEBUG"):
            print(f"[DEBUG] {func.__name__}")
        return func(*args, **kwargs)
    return wrapper
```

### 方法装饰器（处理 self）

> **💡 思路：装饰方法时，wrapper 的第一个参数就是 self，正常透传即可**

```python
def validate_positive(method):
    @wraps(method)
    def wrapper(self, value, *args, **kwargs):
        if value <= 0:
            raise ValueError(f"必须为正数: {value}")
        return method(self, value, *args, **kwargs)
    return wrapper

class BankAccount:
    @validate_positive
    def deposit(self, amount):
        self._balance += amount
```

## 总结：装饰器的思维模型

| 思维原则 | 核心问题 | 对应知识点 |
|---------|---------|-----------|
| 分离关注点 | "这个逻辑跟核心业务有关吗？" | 所有装饰器 |
| 开闭原则 | "能不改原函数就加功能吗？" | 装饰器基本概念 |
| 可配置 | "不同场景需要不同行为吗？" | 带参数装饰器 |
| 可组合 | "多个增强能叠加吗？" | 装饰器叠加 |
| 透明性 | "装饰后还能看到原函数信息吗？" | @wraps |
| 注册 | "需要记录谁有什么能力吗？" | 注册模式 |

**核心思路：装饰器的本质是"在不修改原代码的前提下，给函数/类动态添加功能"。它是 Python 实现 AOP（面向切面编程）的主要工具。**

**记住一个判断标准：如果一段逻辑在多个函数中重复出现，且与核心业务无关 → 考虑用装饰器提取出来。**

## 练习题（工程化版）

### 练习 1（基础）

写 `@timer` 装饰器，打印函数执行时间。

<details><summary>答案</summary>

```python
import time
from functools import wraps

def timer(func):
    @wraps(func)
    def wrapper(*args, **kwargs):
        start = time.perf_counter()
        result = func(*args, **kwargs)
        print(f"{func.__name__}: {time.perf_counter() - start:.4f}s")
        return result
    return wrapper
```
</details>

### 练习 2（带参数）

写 `@retry(times=3)` 装饰器。

<details><summary>答案</summary>

```python
def retry(times=3):
    def deco(func):
        @wraps(func)
        def wrapper(*args, **kwargs):
            for i in range(times):
                try:
                    return func(*args, **kwargs)
                except Exception:
                    if i == times - 1:
                        raise
        return wrapper
    return deco
```
</details>

### 练习 3（验证）

写 `@validate_types` 装饰器，根据类型注解验证参数。

<details><summary>答案</summary>

```python
import inspect
from functools import wraps

def validate_types(func):
    sig = inspect.signature(func)
    @wraps(func)
    def wrapper(*args, **kwargs):
        bound = sig.bind(*args, **kwargs)
        for name, value in bound.arguments.items():
            expected = sig.parameters[name].annotation
            if expected != inspect.Parameter.empty:
                if not isinstance(value, expected):
                    raise TypeError(f"{name} expects {expected}, got {type(value)}")
        return func(*args, **kwargs)
    return wrapper

@validate_types
def add(a: int, b: int) -> int:
    return a + b

add(1, 2)        # ✅
# add(1, "2")    # ❌ TypeError
```
</details>

### 练习 4（综合）

写一个 `@memoize` 装饰器，缓存函数结果（不限大小）。

<details><summary>答案</summary>

```python
def memoize(func):
    cache = {}
    @wraps(func)
    def wrapper(*args, **kwargs):
        key = (args, frozenset(kwargs.items()))
        if key not in cache:
            cache[key] = func(*args, **kwargs)
        return cache[key]
    wrapper.cache = cache
    return wrapper
```

实际项目用 `functools.lru_cache(maxsize=...)`。
</details>
